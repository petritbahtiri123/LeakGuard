# Phase 17D Corrupted/Large-File Fuzz Automation

## Scope

Phase 17D adds deterministic automation for corrupted, malformed, and oversized file paths. It is test coverage only: no product features, detector rule changes, permissions, dependencies, backend calls, telemetry, remote verification, or large checked-in binary fixtures.

All fuzz fixtures are generated at test runtime from a deterministic seed: `phase-17d-deterministic-seed-v1`. Any future random fuzz must use a fixed seed and include that seed in failure output.

## Fixture Generation

The deterministic generator in `tests/fixtures/file_fuzz_fixtures.js` creates small or runtime-generated fixtures for:

- malformed PDF, truncated PDF, image-only/no-text PDF, and oversized text PDF
- malformed DOCX ZIP, DOCX missing `word/document.xml`, DOCX with weird XML entities, and oversized DOCX extracted text
- malformed XLSX ZIP, XLSX missing workbook/shared string/sheet pieces, XLSX with formula/comment/hidden-sheet/custom XML raw markers, and oversized XLSX extracted text
- corrupt PNG/JPG/WEBP metadata paths and oversized image metadata path
- large `.env`, `.json`, `.log`, `.md`, and source text files
- unsupported legacy Office extensions `.doc`, `.docm`, `.xls`, and `.xlsm`

Fixtures embed synthetic markers only. The raw marker sweep checks that raw markers do not appear in safe extractor state, scanner reports, protected-site captures, downloads, or rebuilt artifacts.

## Extractor Fuzz Coverage

`tests/file_extractors.test.js` exercises the deterministic fixture set through `prepareFileExtractionAsync`.

Expected behavior:

- malformed inputs never throw uncaught exceptions
- unsupported or unsafe inputs return safe failure or unsupported states
- fail closed outputs have empty extracted text and `safeForScan: false`
- oversized inputs respect configured PDF/DOCX/XLSX text limits
- scanner redaction/report paths do not retain raw markers for safe metadata/text cases
- bounded runtime is asserted, with deterministic seed included in failure output

## Rebuilt-Output Fuzz Coverage

The existing redactor tests cover regenerated output contracts:

- PDF/DOCX/XLSX generators do not copy original PDF streams or OOXML parts
- raw marker input does not survive generated bytes or extracted text after redaction
- generated output remains parseable by the existing extractor when expected
- oversized sanitized text is bounded and marked truncated
- unsafe extraction states do not create rebuilt outputs

Phase 17D keeps the invariant that empty or unsafe sanitized text produces no rebuilt file or a safe failure. Bounded/truncated rebuilt output remains eligible for scanner download but protected-site handoff must fall back or fail closed where required.

## Protected-Site Fuzz Coverage

Chrome full QA covers protected-site file picker fail-closed paths for malformed PDF, malformed DOCX, malformed XLSX, image-only PDF, and unsupported `.doc`, `.docm`, `.xls`, and `.xlsm` uploads. Provider page text, uploads, submissions, input files, modal text, panel text, and overlay text are swept for raw markers.

Oversized safe protected-site files are expected to produce a sanitized fallback or fail closed with no raw upload. Firefox and Edge keep smoke coverage only for this phase; full protected-site fuzz handoff remains Chrome full QA.

## Scanner Fuzz Coverage

Scanner browser QA covers malformed PDF, unsupported/image-only DOCX, image-only PDF, empty PDF, encrypted PDF, malformed PDF, corrupt image metadata fallback, scanner redacted downloads, and sanitized JSON report downloads. The scanner path must not offer unsafe regenerated downloads for malformed or empty structured files.

Large text/source file handling is covered by extractor/unit tests and the file-extraction benchmark to avoid making normal browser smoke runs unnecessarily slow.

## Performance Guardrails

`tests/performance/file-extraction-pipeline-benchmark.mjs` covers deterministic large and malformed file paths without heavy dependencies. Phase 17D extends the intended benchmark surface to large text, large PDF extraction, large DOCX extraction, large XLSX extraction, and the stable image metadata/OCR probe path where practical.

Budgets stay realistic and deterministic. Normal `npm test` remains bounded; heavier browser and package checks run through explicit QA/release commands.

## Raw Marker Sweep

Raw marker sweep coverage includes:

- generated fixture outputs
- extractor safe states
- scanner previews, downloads, and JSON reports
- protected-site page text, uploads, submissions, input files, modal text, panel text, and overlay text
- release/package artifacts only during release/package validation, not every tiny unit test

## Readiness

Phase 17D is ready when deterministic extractor fuzz, rebuilt-output fuzz, protected-site fuzz, scanner fuzz, raw marker sweeps, performance guardrails, browser QA, package generation, and `git diff --check` all pass.

Remaining limitations should be documented rather than faked. Live-provider fuzz selectors are out of scope; synthetic protected-site harnesses and adapter contract guardrails are preferred.
